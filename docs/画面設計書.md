### 3.7 デザインの方向性
- ユーザが競輪予想を「直感的に」「楽しく」「効率よく」行える体験を提供すること。
- 視認性が高く、情報量が多くてもストレスにならないレイアウト
- 競走選手やレースの特徴が一目で理解できるビジュアル
- 予想プロセスをガイドするインタラクション
- ワクワク感やゲーム性を感じられるUI
- 映画「男はつらいよ」の時代感、ビール瓶や赤鉛筆、畳の上で予想を広げるような泥臭くも人情味があって熱い**「昭和レトロな賭
場のオジサン」**スタイル

### 3.8 デザイン要素
- フォント：視認性重視
- アイコン：選手の脚質や展開を直感的に理解できるもの
- アニメーション：予想プロセスやレース展開の理解を助ける軽い動き

#### 4.1.5 データ取得条件

```sql
-- 当日開催の競輪場を取得
SELECT rs.*, p.program_type, p.kaisai_type
FROM race_schedules rs
JOIN programs p ON p.race_schedule_id = rs.id
WHERE p.kaisai_date = {当日日付(JST)}
ORDER BY rs.grade ASC, rs.jyo_cd ASC;
```

> グレード優先度: GP > G1 > G2 > G3 > F1 > F2 の順でソート

```sql
-- 当該競輪場・当日のレース一覧を取得

FROM races r

JOIN race_schedules rs ON rs.id = p.race_schedule_id
WHERE p.kaisai_date = {対象日付}
  AND rs.jyo_cd = {競輪場コード}
ORDER BY r.race_no ASC;
```

## 6. データフェッチ設計

### 6.1 各画面のデータ取得方式

| 画面 | 取得方式 | キャッシュ | 備考 |
|------|---------|----------|------|

| `/api/race/recent` | GET | `race_id` | 出走表（直近成績）取得 |

### A.4 動作仕様

| No. | 仕様 | 内容 |
|-----|------|------|
| 1 | 実行中制御 | ボタン押下中は全ボタンを非活性化しスピナーを表示 |
| 2 | 同時実行禁止 | 複数処理の同時実行は禁止（実行中フラグで制御） |
| 3 | 日付パラメータ | 日付ピッカーに値がある場合、`target_date` をリクエストに付与。空欄の場合は省略 |
| 4 | 削除処理の日付 | `step: "cleanup"` は日付ピッカーの値を無視する旨を UI 上に注記 |
| 5 | ログ表示 | 完了時: ✓ アイコン + メッセージ / エラー時: ✗ アイコン + エラーメッセージ |
| 6 | job_run_id | 実行ログに `job_run_id` を表示し、失敗時にトラブルシュートを可能にする |
| 7 | 前提バリデーション | 依存テーブルにデータがない状態で個別実行した場合、409 エラーメッセージを表示 |
