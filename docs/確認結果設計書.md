## 1. プロジェクト概要

### 1.1 システム目的
競輪のレース情報をスクレイピングし、AI（Gemini API）を活用して予想を生成するシステム。

### 1.2 技術スタック

| レイヤ | 技術 |
|--------|------|
| Frontend | Next.js 16.1.6, React 19.2.3, TypeScript |
| Backend | Node.js, Next.js App Router, TypeScript |
| Database | Supabase (PostgreSQL) with RLS |
| Scraping | Cheerio, Axios, Puppeteer |
| AI | Gemini API (Google) |
| Deployment | Vercel |
| CI/CD | GitHub Actions |

### 1.3 主要機能
1. **スクレイピング** (netkeiba.com)
   - レース日程・プログラムの自動取得
   - 出走表3種（基本情報・直近成績・対戦表）の取得
   - 定時実行（AM 05:15 JST）+ 手動実行

2. **AI予想**
   - Gemini APIによる自動予想生成
   - スクレイピング完了後に自動実行
   - 管理画面からの手動実行も可能

3. **管理画面** (`/admin`)
   - スクレイピング/AI予想/削除の手動起動
   - ジョブ実行履歴（リアルタイムポーリング）
   - 対象日付指定機能

## 4. スクレイピング機能

### 4.1 スクレイパー構成

| ファイル名 | スクレイピング対象 |
|-----------|-------------------|
| `raceScheduleScraper.ts` | レース日程（GP〜G3）＋本日開催一覧（F1/F2補完） |
| `raceProgramScraper.ts` | 開催情報・レース一覧＋開催区分（カレンダー） |
| `raceEntryScraper.ts` | 出走表（基本情報・並び予想） |
| `raceRecentResultScraper.ts` | 出走表（直近成績：今節・直近1〜3開催） |
| `raceMatchResultScraper.ts` | 出走表（対戦表） |

### 4.2 実行フロー

```
STEP 0: ジョブロック取得（job_runs テーブルで排他制御）
  └─> checkAlreadyRunning(): job_runs.status='running' のレコードを確認

STEP 1: レース日程スクレイピング
  ├─> scrapeGradeSchedules(): /race/schedule/?kaisai_year={YYYY}&kaisai_month={MM}
  └─> scrapeTodayRaceList(): /?kaisai_date={YYYYMMDD}（F1/F2補完）
      └─> race_schedules へ UPSERT

STEP 2: 当日開催競輪場の特定
  └─> getActiveSchedulesByDate(): start_date <= target_date <= end_date

STEP 3: レースプログラムスクレイピング（各競輪場）
  ├─> scrapeKaisaiTypeMap(): 開催区分（カレンダー）
  ├─> scrapeProgramByJyo(): /db/race_program/?kaisai_group_id={YYYYMMDD}{JYO_CD}
  ├─> programs へ UPSERT
  └─> races へ UPSERT（program_idはUPSERT後取得）

STEP 4: 当日レース一覧の特定
  └─> getRacesByDate(): kaisai_date = target_date

STEP 5: 出走表3種スクレイピング（各レース）
  ├─> scrapeEntry(): /race/entry/?race_id={race_id}
  ├─> scrapeRecentResults(): /race/entry/results.html?race_id={race_id}
  ├─> scrapeMatchResults(): /race/match_list/?race_id={race_id}
  └─> 3種を並列実行 → 各テーブルへ UPSERT

STEP 6: AI予想実行（skipPrediction=falseの場合）
  └─> runPrediction(): Gemini APIで予想生成

STEP 7: ジョブロック解放
  └─> finishJobRun(): statusをsuccess/failedに更新
```

### 4.3 レートリミット対策
- 各リクエスト間に500msのウェイト（環境変数`SCRAPE_DELAY_MS`で制御可能）
- HTTP 4xx/5xx: 指数バックオフで最大3回リトライ（1秒→2秒→4秒）
- タイムアウト（>10秒）: リトライなし、即時エラー

### 4.4 エラーハンドリング

| エラー種別 | 対処方針 |
|-----------|---------|
| HTTP 4xx / 5xx | 3回リトライ（指数バックオフ）後、STEPを失敗としてログ記録 |
| タイムアウト（>10秒） | リトライなし、STEPを失敗としてログ記録 |
| HTML 解析エラー | 当該レース/選手をスキップ、次のデータ処理へ継続 |
| DB 書き込みエラー | 当該レースをスキップ、次のデータ処理へ継続 |
| STEP 1失敗 | 後続のSTEP 2〜5をスキップ |
| STEP 3失敗（一部競輪場） | その競輪場のSTEP 5をスキップして他の競輪場は継続 |
| STEP 5失敗（一部レース） | そのレースをスキップして次のレースへ継続 |

## 5. AI予想機能

### 5.1 アーキテクチャ

```
出走表データ（DB）
    ├─ race_entries: 基本情報
    ├─ race_recent_results: 直近成績
    └─ race_match_results: 対戦表
          ↓
    formatRaceData(): レースデータを整形してプロンプト用にする
          ↓
    fullPrompt = prompt + レースデータ + 出力要求
          ↓
    callAI(): Gemini API (gemini-2.0-flash-exp)
          ↓
    parseAIResponse(): セクション1〜7にパース
          ↓
    race_predictions へ UPSERT
```

### 5.2 使用LLM

| 項目 | 設定 |
|------|------|
| モデル | `gemini-2.0-flash-exp`（環境変数 `GEMINI_MODEL`） |
| API | Gemini API (Google) |
| APIキー | 環境変数 `GEMINI_API_KEY` |
| 最大出力トークン | 8192 |
| Temperature | 0.7 |

### 5.3 予想プロンプト

**ファイル**: `docs/prediction-prompt.txt`

**セクション構成**:
- セクション1: 自信度
- セクション2: 展開予想
- セクション3: ライン別評価
- セクション4: 本命シナリオ
- セクション5: 中穴シナリオ
- セクション6: 推奨買い目
- セクション7: 寅次郎の一言

### 5.4 エラーハンドリング

| エラー種別 | 対処方針 |
|-----------|---------|
| 401 Unauthorized | 「Invalid API key」エラー |
| 404 Not Found | 「Model not found」エラー |
| 429 Rate Limit | 指数バックオフで最大3回リトライ |
| JSONパースエラー | 「Invalid JSON」エラー |
| その他APIエラー | エラー詳細を記録して次のレースへ継続 |
| レースごとのエラー | 次のレースへ継続（全体を中断しない） |

### 5.5 レートリミット対策
- 各レース処理後に3秒のディレイ（環境変数で制御可能）
- APIコールのリトライロジック（最大3回）

## 6. APIエンドポイント

### 6.1 管理API (`/api/admin/*`)

#### 6.1.1 スクレイピングトリガー

**Endpoint**: `POST /api/admin/trigger`

**Headers**:
- `x-admin-api-key: {ADMIN_API_KEY}`

**Body (JSON)**:
```json
{
  "workflow": "scrape" | "cleanup" | "prediction",
  "step": "all" | "schedule" | "program" | "entry" | "prediction",
  "target_date": "YYYY-MM-DD"
}
```

**機能**: GitHub Actionsのworkflow_dispatchを起動する軽量プロキシ

**Responses**:
- `202 Accepted` - GitHub Actionsにdispatch送信済み
- `400 Bad Request` - バリデーションエラー
- `401 Unauthorized` - APIキー認証失敗
- `500 Internal Server Error` - サーバーエラー

#### 6.1.2 AI予想実行

**Endpoint**: `POST /api/admin/prediction`

**Headers**:
- `x-admin-api-key: {ADMIN_API_KEY}`

**Body (JSON)**:
```json
{
  "target_date": "YYYY-MM-DD"
}
```

**機能**: AI予想の手動実行

**Responses**:
- `200 OK` - 予想実行完了
- `400 Bad Request` - バリデーションエラー
- `401 Unauthorized` - APIキー認証失敗
- `500 Internal Server Error` - サーバーエラー

#### 6.1.3 ジョブ履歴取得

**Endpoint**: `GET /api/admin/job-runs`

**Headers**:
- `x-admin-api-key: {ADMIN_API_KEY}`

**Query**:
- `limit`: 最大取得件数（default: 20, max: 50）

**Responses**:
- `200 { runs: JobRun[], hasRecentFailure: boolean }`
- `401 Unauthorized` - APIキー認証失敗
- `500 Internal Server Error` - サーバーエラー

### 6.2 ヘルスチェック

**Endpoint**: `GET /api/health`

**機能**: 環境変数が正しく設定されているかを確認

**Response**:

## 7. 管理画面

### 7.1 概要

**URL**: `/admin`

**機能**:
- スクレイピング手動実行（全処理・個別実行）
- AI予想手動実行
- 削除処理実行
- ジョブ実行履歴のリアルタイム表示（3秒ごとにポーリング）
- 対象日付指定（日付ピッカー）
- 直近24時間以内の失敗ジョブ警告バナー

### 7.2 UI構成

```
┌─────────────────────────────────────────┐
│  競輪予想 管理画面                        │
├─────────────────────────────────────────┤
│  ⚠️ 直近24時間以内にfailedのジョブがあります │
├─────────────────────────────────────────┤
│  ■ スクレイピング実行                      │
│  対象日付: [____年__月__日▼]              │
│  [全処理実行] [レース日程] [レースプログラム] │
│  [出走表] [削除処理]                      │
├─────────────────────────────────────────┤
│  ■ AI予想実行                             │
│  [AI予想実行（対象日付: YYYY-MM-DD）]       │
├─────────────────────────────────────────┤
│  ■ ジョブ実行履歴                          │

└─────────────────────────────────────────┘
```

### 7.3 日付ピッカー仕様
- 最大値: JST当日日付（未来日付選択不可）
- 空欄の場合: サーバー側で当日日付をデフォルト適用
- `step: "cleanup"` 時は日付ピッカーの値を無視

## 8. ジョブ管理

### 8.1 ジョブ種別

| ジョブ種別 | 説明 |
|----------|------|
| `cron_scrape` | 定時スクレイピング |
| `cron_cleanup` | 定時データ削除 |
| `admin_scrape` | 手動スクレイピング |
| `admin_prediction` | 手動AI予想 |

### 8.2 ステップ定義

| ステップ | 説明 |
|---------|------|
| `all` | 全処理実行（日程→プログラム→出走表→予想） |
| `schedule` | レース日程のみ |
| `program` | レースプログラムのみ |
| `entry` | 出走表のみ |
| `prediction` | AI予想のみ |
| `cleanup` | データ削除のみ |

### 8.3 ジョブステータス

| ステータス | 説明 |
|---------|------|
| `running` | 実行中 |
| `success` | 成功 |
| `failed` | 失敗 |
| `skipped` | スキップ |

### 8.4 排他制御
- ジョブ実行前に `job_runs` テーブルで `status='running'` のレコードを確認
- 同一ジョブ種別の実行中ジョブが存在する場合、新規ジョブをスキップ
- ジョブ実行完了時に `finished_at` を設定し `status` を更新

## 11. セキュリティ

### 11.1 認証
- 管理API: `x-admin-api-key` ヘッダーによる認証

### 11.3 APIキー管理
- Supabase Service Roleキーはサーバーサイドのみで使用
- クライアント側には絶対に露出させない

## 14. まとめ

本プロジェクトは以下の要件に基づき設計・実装されていることが確認されました：

1. **スクレイピング機能**: netkeiba.comから必要なデータを取得し、Supabaseに保存
2. **AI予想機能**: Gemini APIを使用してレース予想を生成
3. **管理機能**: 手動実行・ジョブ履歴表示・ヘルスチェック
4. **定時実行**: 毎日AM05:15にスクレイピング、AM05:00にデータ削除

アーキテクチャはクリーンで、以下のベストプラクティスに従っている：
- レイヤードアーキテクチャ（Scrapers → Repositories → Services → API）
- 型安全（TypeScript）
- エラーハンドリング（リトライ・ログ記録・ジョブ履歴）
- 排他制御（ジョブ実行管理）
- レートリミット対策

今後は以下の改善が推奨されます：
- 全テーブルのRLSポリシー設定
- Cron APIの実装
- ユーザー向け予想結果表示UIの実装
