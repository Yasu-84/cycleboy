## 2. スケジュール処理設計

### 2.1 定時実行一覧

| 処理名             | 実行時刻 (JST) | Vercel Cron (UTC) | 概要                        |
| ------------------ | -------------- | ----------------- | --------------------------- |
| データ削除処理     | AM 05:00       | `0 20 * * *`    | 7日以上経過したデータを削除 |
| スクレイピング処理 | AM 05:15       | `15 20 * * *`   | 当月日程・当日出走表を取得  |

> **注意**: Vercel Cron は UTC で設定するため JST との差分 (-9h) を考慮すること。
> AM05:00 JST = UTC 前日 20:00 / AM05:15 JST = UTC 前日 20:15
> DBの `TIMESTAMPTZ` は UTC で保存し、業務日判定（当日開催判定）は JST で行う。

### 2.2 vercel.json Cron 設定

```json
{
  "crons": [
    {
      "path": "/api/cron/cleanup",
      "schedule": "0 20 * * *"
    },
    {
      "path": "/api/cron/scrape",
      "schedule": "15 20 * * *"
    }
  ]
}
```

## 3. 処理フロー

### 3.1 データ削除処理フロー（AM 05:00）

```
/api/cron/cleanup
    │
    ├── 削除基準日時を算出（実行時刻 - 7日）
    │
    ├── 1. race_match_results  ← created_at < 基準日時 を削除
    ├── 2. race_recent_results ← created_at < 基準日時 を削除
    ├── 3. race_entries        ← created_at < 基準日時 を削除
    ├── 4. races               ← created_at < 基準日時 を削除
    ├── 5. programs            ← created_at < 基準日時 を削除
    └── 6. race_schedules      ← created_at < 基準日時 を削除
```

> **外部キー制約（RESTRICT）** があるため、子テーブルから順に削除する。
> CASCADE を指定しないことで、意図しない連鎖削除を防止する。

### 3.2 スクレイピング処理フロー（AM 05:15）

```
/api/cron/scrape または /api/admin/scrape（手動）
    │
    ├── [STEP 0] ジョブロック取得
    │       処理: 同時実行防止のためロックを取得（例: pg_advisory_lock）
    │       結果: 取得失敗時は `409 Conflict` を返して終了
    │
    ├── [STEP 1] レース日程スクレイピング
    │       URL1: /race/schedule/?kaisai_year={YYYY}&kaisai_month={MM}
    │             → グレードレース（GP〜G3）の開催日程を取得
    │       URL2: /race/race_list/
    │             → 本日の全開催一覧（F1/F2 含む）を取得
    │       対象: 実行日の当月（YYYY-MM）
    │       処理: 取得データを race_schedules へ UPSERT
    │
    ├── [STEP 2] 当日開催競輪場の特定
    │       処理: race_schedules から start_date <= 実行日 <= end_date のレコードを取得
    │             → 開催中の jyo_cd リストを取得
    │       補足: STEP 1 で F1/F2 も含めて取得済みのため全グレードが対象
    │
    ├── [STEP 3] レースプログラムスクレイピング（for each jyo_cd）
    │       URL1: /race/course/?jyo_cd={JYO_CD}
    │             → 当日開催の開催情報・レース一覧を取得
    │       URL2: /race/course/calendar.html?kaisai_year=...&kaisai_month=...&jyo_cd={JYO_CD}&kaisai_type=1
    │             → 開催区分（モーニング/ナイター/ミッドナイト/ガールズ）を取得
    │       処理: 取得データを programs・races へ UPSERT
    │       紐付: race_schedules を jyo_cd + 日付範囲(start_date <= kaisai_date <= end_date)で
    │             検索し、該当する race_schedule_id を FK として設定する
    │
    ├── [STEP 4] 当日レース一覧の特定
    │       処理: programs・races から kaisai_date = 実行日のレースを収集
    │             → race_id（ネットケイリン形式）リストを取得
    │
    └── [STEP 5] 出走表スクレイピング（for each jyo_cd 単位で順次実行）
            ※ 競輪場単位で処理を分割（タイムアウト対策）
            ├── 出走表(基本情報・並び予想)
            │       URL: /race/entry/?race_id={race_id}
            │       処理: 取得データを race_entries へ UPSERT
            │             並び予想は formation_prediction (JSONB) カラムに格納
            │
            ├── 出走表(直近成績)
            │       URL: /race/entry/results.html?race_id={race_id}
            │       処理: 取得データを race_recent_results へ UPSERT
            │
            └── 出走表(対戦表)
                    URL: /race/match_list/?race_id={race_id}
                    処理: 取得データを race_match_results へ UPSERT

                    └── [STEP 6] ジョブロック解放
                      処理: 成否にかかわらず finally でロックを解放
```

> **対象日付 (target_date)**：
>
> - スクレイピング処理は「対象日付」を基準に実行する。対象日付は API リクエストの `target_date` パラメータで指定する（省略時は実行時点の JST 当日日付を使用）。
> - STEP 1 では `target_date` の年月を kaisai_year / kaisai_month に使用する。
> - STEP 2 / STEP 4 では `target_date` を「実行日」として日付フィルタリングを行う。
> - `step: "cleanup"` 時は `target_date` を無視し、常に「実行時 JST 日付 − 7 日」を閾値とする。

> **レートリミット対策**：各リクエスト間に 500ms のウェイトを挿入する。

> **タイムアウト対策**：Vercel の関数タイムアウト（Pro: 最大300秒）を考慮し、以下の方針を採用する。
>
> - route.ts で `export const maxDuration = 300;` を設定（Vercel Pro プラン前提）
> - 当日の開催が多い場合（5場 × 12R × 3種 = 最大180リクエスト、500ms/req で約90秒）でも
>   300秒以内に収まることを確認する
> - 万が一タイムアウトが発生する場合は、管理画面から競輪場単位（step=entry）で個別実行する

> **トランザクション方針**：
>
> - レース単位でトランザクションを開始し、`race_entries` / `race_recent_results` / `race_match_results` の
>   3種をまとめてコミットする
> - 途中失敗時は当該レースのみロールバックし、次レースの処理を継続する
> - 全書き込みは UPSERT で実装し、再実行時も同一結果になる冪等性を担保する

### 4.2 テーブル定義

#### 4.2.1 race_schedules（レース日程）

| カラム名    | 型          | 制約                          | 説明                                 |
| ----------- | ----------- | ----------------------------- | ------------------------------------ |
| id          | UUID        | PK, DEFAULT gen_random_uuid() | 内部識別子                           |
| jyo_cd      | TEXT        | NOT NULL                      | 競輪場コード（例: "35"）             |
| jyo_name    | TEXT        | NOT NULL                      | 競輪場名（例: "平塚"）               |
| grade       | TEXT        | NOT NULL                      | グレード（GP/G1/G2/G3/F1/F2）        |
| kaisai_name | TEXT        | NOT NULL                      | 開催名（例: "KEIRINグランプリ2025"） |
| start_date  | DATE        | NOT NULL                      | 初日                                 |
| end_date    | DATE        | NOT NULL                      | 最終日                               |
| created_at  | TIMESTAMPTZ | NOT NULL, DEFAULT now()       | 登録日時                             |

**ユニーク制約**: `(jyo_cd, start_date)` — 同競輪場・同初日の重複を防ぐ

---

#### 4.2.2 programs（開催）

| カラム名         | 型          | 制約                              | 説明                                     |
| ---------------- | ----------- | --------------------------------- | ---------------------------------------- |
| id               | UUID        | PK, DEFAULT gen_random_uuid()     | 内部識別子（プログラムID）               |
| race_schedule_id | UUID        | FK → race_schedules.id, RESTRICT | 所属するレース日程ID                     |
| kaisai_date      | DATE        | NOT NULL                          | 開催日                                   |
| grade            | TEXT        | NOT NULL                          | グレード（GP/G1/G2/G3/F1/F2）            |
| kaisai_type      | TEXT[]      | NULLABLE                          | 開催区分（例: ["ナイター", "ガールズ"]） |
| program_type     | TEXT        | NOT NULL                          | プログラム種別（初日/2日目/最終日 等）   |
| created_at       | TIMESTAMPTZ | NOT NULL, DEFAULT now()           | 登録日時                                 |

**ユニーク制約**: `(race_schedule_id, kaisai_date)` — 同開催・同日の重複を防ぐ

> **備考**:
>
> - `kaisai_type` は NULL 可（通常開催の場合は設定なし）
> - 最大2種類の開催区分が設定されることがある（例: `["モーニング", "ガールズ"]`）

---

#### 4.2.3 races（レースプログラム）

| カラム名         | 型          | 制約                          | 説明                                                       |
| ---------------- | ----------- | ----------------------------- | ---------------------------------------------------------- |
| id               | UUID        | PK, DEFAULT gen_random_uuid() | 内部識別子（レースID）                                     |
| program_id       | UUID        | FK → programs.id, RESTRICT   | 所属するプログラムID                                       |
| netkeiba_race_id | TEXT        | UNIQUE, NOT NULL              | ネットケイリンのレースID（{YYYYMMDD}{JYO_CD}{RaceNo:2桁}） |
| race_no          | INTEGER     | NOT NULL                      | レース番号（1〜24）                                        |
| race_title       | TEXT        | NOT NULL                      | レースタイトル（例: "Ｓ級 初日特別選抜"）                  |
| car_count        | INTEGER     | NOT NULL                      | 車立数（4〜9）                                             |
| departure_time   | TIME        | NOT NULL                      | 発走時刻                                                   |
| deadline_time    | TIME        | NOT NULL                      | 締切時刻                                                   |
| created_at       | TIMESTAMPTZ | NOT NULL, DEFAULT now()       | 登録日時                                                   |

> **netkeiba_race_id 生成ルール**: `{YYYYMMDD}` + `{JYO_CD（ゼロ埋め2桁）}` + `{レース番号（ゼロ埋め2桁）}`
> 例: 2026年3月3日 大垣(44) 第12レース → `202603034412`
> **注意**: JYO_CD が1桁の場合（例: 函館=5）はゼロ埋めして `05` とする。
> 実際にはネットケイリンの URL から取得した race_id をそのまま使用するため、
> 本システムで独自生成はせず、HTML の `href` 属性から抽出した値を使用する。

```json
  ]
}
```

> **備考**: `formation_prediction` は1レースにつき1つのデータだが、同一レースの全エントリレコードに同じ値を格納する。

### 4.3 インデックス設計

| テーブル            | インデックス                          | 用途                     |
| ------------------- | ------------------------------------- | ------------------------ |
| race_schedules      | `(jyo_cd, start_date, end_date)`    | 当日開催場の抽出         |
| programs            | `(kaisai_date)`                     | 当日プログラム検索       |
| races               | `(program_id, race_no)`             | プログラム配下レース検索 |

### 6.2 手動起動エンドポイント

#### POST /api/admin/scrape

| 項目             | 内容                                                                                               |
| ---------------- | -------------------------------------------------------------------------------------------------- |
| 認証             | `x-admin-api-key: {ADMIN_API_KEY}` ヘッダー検証（不正アクセス防止）                              |
| リクエストボディ | `{ "step": "all" \| "schedule" \| "program" \| "entry" \| "cleanup", "target_date"?: "YYYY-MM-DD" }` |
| 処理             | `step` に応じた処理のみを実行。`target_date` を対象日付として使用（省略時は JST 当日）         |
| 正常レスポンス   | `200 { "status": "ok", "message": "..." }`                                                       |
| 異常レスポンス   | `401 { "status": "error", "message": "Unauthorized" }`（認証失敗時）                             |
|                  | `409 { "status": "error", "message": "Job already running" }`（排他制御）                        |
|                  | `409 { "status": "error", "message": "Prerequisite data missing" }`（前提データ不足時）          |
|                  | `500 { "status": "error", "message": "..." }`（処理失敗時）                                      |

| `step` 値  | 実行内容                                    | 前提条件                              |
| ------------ | ------------------------------------------- | ------------------------------------- |
| `all`      | スクレイピング全処理（STEP 1〜5）を順次実行 | なし                                  |
| `schedule` | STEP 1 のみ（レース日程）                   | なし                                  |
| `program`  | STEP 3 のみ（レースプログラム）             | race_schedules にデータが存在すること |
| `entry`    | STEP 5 のみ（出走表3種）                    | races にデータが存在すること          |
| `cleanup`  | 削除処理のみ                                | なし                                  |

> **前提条件バリデーション**: `program` や `entry` を個別実行する際、依存テーブルにデータがない場合は `409 Conflict` を返し、管理画面にエラーメッセージ「先にレース日程を取得してください」等を表示する。

> **target_date のバリデーション**: `YYYY-MM-DD` 形式であること、および `target_date` が未来日付でないことをサーバーサイドで検証する。不正な値の場合は `400 { "status": "error", "message": "Invalid target_date" }` を返す。

## 8.1 実行制御設計（排他・再実行）

- 排他制御は DB ロックで実装し、同時実行時は後続要求を `409 Job already running` で即時終了
- ロックキーは `cron_scrape` / `cron_cleanup` / `admin_scrape` を分離し、cleanup と scrape の衝突を防止
- 手動実行は同一 `step` の二重実行を禁止し、未完了ジョブがある場合は管理画面で警告表示
- ジョブ失敗時は `job_runs.status=failed` とし、次回実行は同一UPSERTで再実行可能にする

## 9. エラーハンドリング設計

### 9.1 スクレイピングエラーの扱い

| エラー種別            | 対処方針                                                                 |
| --------------------- | ------------------------------------------------------------------------ |
| HTTP 4xx / 5xx        | 3回リトライ（指数バックオフ: 1秒→2秒→4秒）後、STEPを失敗としてログ記録 |
| タイムアウト（>10秒） | リトライなし、STEPを失敗としてログ記録                                   |
| HTML 解析エラー       | 当該レース/選手をスキップ、次のデータ処理へ継続                          |
| DB 書き込みエラー     | 当該レースをスキップ、次のデータ処理へ継続                               |

### 9.2 処理継続方針

- **STEP 1（レース日程）失敗**: 後続の STEP 2〜5 をスキップし処理終了
- **STEP 3（レースプログラム）の一部競輪場失敗**: その競輪場の STEP 5 をスキップして他の競輪場は継続
- **STEP 5（出走表）の一部レース失敗**: そのレースをスキップして次のレースへ継続

### 9.3 環境変数

| 変数名                        | 説明                                                               |
| ----------------------------- | ------------------------------------------------------------------ |
| `SUPABASE_URL`              | Supabase プロジェクト URL                                          |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase Service Role キー（RLS バイパス用。INSERT/DELETE に必要） |
| `CRON_SECRET`               | Vercel Cron エンドポイントの認証トークン                           |
| `ADMIN_API_KEY`             | 管理エンドポイント（/api/admin/*）の認証キー                       |
| `SCRAPE_DELAY_MS`           | スクレイピング間隔（デフォルト: 500ms）                            |

### 9.4 監視・運用設計

- すべての実行に `job_run_id`（UUID）を採番し、APIレスポンス・サーバーログ・DB履歴を相関させる
- `/admin` の実行ログは `job_runs` を参照し、詳細エラーは `job_errors` を参照して表示する
- 直近24時間で `failed` が1件以上の場合は管理画面に警告バナーを表示する
- `job_errors.detail` に対象URL・HTTPステータス・race_id を保持し、再現調査を容易にする

### 9.5 セキュリティ設計（RLS/鍵管理）

- `SUPABASE_SERVICE_ROLE_KEY` はサーバーサイド（Route Handler / Server Action）のみで使用し、クライアントへ露出しない
- クライアント側 Supabase 利用が必要な将来フェーズでは `anon` キーを別管理し、用途を分離する
- `/api/admin/*` は `ADMIN_API_KEY` 必須、`/api/cron/*` は `CRON_SECRET` 必須で運用する
- 管理画面でキー値を表示・ログ出力しない（マスクのみ）
