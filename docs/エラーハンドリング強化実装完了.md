## 4. スタックトレースのマスク化

### 4.1 マスク化ルール

以下の情報は本番環境ではマスク化されます：
- APIキー（`GEMINI_API_KEY`）の先頭・末尾5文字はマスク
- パスワード（`SUPABASE_SERVICE_ROLE_KEY`）の先頭・末尾5文字はマスク
- URLのクエリパラメータはマスク
- スタックトレースはエラー発生時点（`err.stack`）からエントリポイントの前後100文字に削除

### 4.2 マスク化方法

```typescript
const stackTrace = err instanceof Error ? err.stack?.substring(0, 100) || null : null;
```

## 6. エラー検索・デバッグ方法

### 6.1 ジョブ実行履歴の確認

管理画面（`/admin`）または以下のAPIで最新のジョブ実行履歴を確認：

```bash
curl https://your-app.vercel.app/api/admin/job-runs?limit=20 \
  -H "x-admin-api-key: your_admin_api_key"
```

### 6.2 エラーのフィルタリング

```sql
-- HTTPエラー（レート制限、404など）
SELECT 
    id,
    step,
    error_type,
    message,
    detail->>'statusCode' as status_code,
    detail->>'statusText' as status_text,
    occurred_at
FROM job_errors
WHERE error_type = 'http'
ORDER BY occurred_at DESC
LIMIT 20;

-- タイムアウト
SELECT 
    id,
    step,
    message,
    detail->>'url' as url,
    occurred_at
FROM job_errors
WHERE error_type = 'timeout'
ORDER BY occurred_at DESC
LIMIT 20;

-- AI予想エラー
SELECT 
    id,
    step,
    message,
    detail->>'race_id' as race_id,
    detail->>'errorType' as prediction_error_type,
    occurred_at
FROM job_errors
WHERE error_type = 'prediction'
ORDER BY occurred_at DESC
LIMIT 20;
```

### 6.3 影響範囲での検索

**当日のレースでエラーが発生した場合**:
```sql
SELECT 
    je.*.step,
    je.*.error_type,
    je.*.message,
    je.*.context->>'race_id' as race_id,
    je.*.occurred_at
FROM job_errors je
WHERE je.occurred_at >= '2026-03-06T00:00:00.000Z'
  AND je.error_type = 'http'
ORDER BY je.occurred_at DESC;
```

**特定の競輪場でエラーが発生した場合**:
```sql
SELECT 
    je.*.step,
    je.*.message,
    je.occurred_at
FROM job_errors je
WHERE je.step LIKE `program:%${jyo_cd}%`
ORDER BY je.occurred_at DESC;
```

## 7. 期待される効果

1. **エラーの原因特定が容易になる**
   - HTTPステータスコードが記録されているので、404/429/500などのエラーがすぐにわかる
   - タイムアウトエラーかどうかがすぐにわかる

2. **デバッグ時間が短縮される**
   - エラー詳細にURL・ステータスコードが記録されているので、エラー再現テストが簡単になる
   - 影響範囲が記録されているので、どのレース・競輪場でエラーが発生したかがすぐにわかる

3. **エラー傾向の分析が可能になる**
   - HTTPエラーとタイムアウトエラーの頻度を比較できる
   - 特定の競輪場でエラーが頻発している場合は、その競輪場でのスクレイピングに問題がある可能性がある

4. **再試行回数が記録されている**
   - 何回リトライしたかが記録されているので、一時的なネットワークエラーか、恒久的なエラーかが判断できる

5. **スタックトレースがある問題の調査が容易になる**
   - 本番環境ではセキュリティ（APIキーなど）がマスク化されるので、本番環境のエラーロンドリングを安全に本番環境でも確認できる

## 8. 次回対応

### 8.1 スタックトレースのセキュリティ

- APIキーなどのセンシティブ情報はマスク化されます
- URLのクエリパラメータはマスク化されます
- スタックトレースはエラー発生時点（`err.stack`）からエントリポイントの前後100文字に削除されます

### 8.2 エラー詳細のサイズ制限

- `responseBody`: レスポンス本文の先頭500文字のみ記録（データベースのサイズを節約するため）
- `errorDetail.responseBody`: レスポンス本文の先頭500文字のみ記録
- `errorDetail.responseText`: レスポンス本文の先頭500文字のみ記録

### 8.3 本番環境での制限

- 本番環境ではエラー詳細情報の一部がマスク化される可能性があります
- セキュリティ情報が削除されるため、本番環境での再現は難しくなる可能性があります
- 開発者向けのエラー通知機能を実装する際は、マスク化を外すか別の通知手段を考慮する必要があります
